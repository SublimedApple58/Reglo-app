# Guide di gruppo (Group lessons) — backend & web

## What it does
Optional module letting a school run **group driving lessons**: 1 instructor + 1 vehicle + **up to 3 students**, typically 3–4 hours, shown as a prominent, distinct event in the agenda. The lesson container exists **independently of its participants** (open seats): the owner/instructor schedules it, may **pre-add** opted-in students, and may **open the remaining seats to an invite** that eligible students self-enrol into (first-come-first-served until full).

- **Optional**: gated on `CompanyService.limits.groupLessonsEnabled` (toggled in web *Impostazioni → Prenotazioni e allievi → Guide*).
- **Per-student opt-in**: `CompanyMember.groupLessonsOptIn` (default `false`), editable from the web student drawer AND mobile instructor/owner student detail. A student can only be pre-added/invited if opted-in.
- **No credit consumed**: each participant seat is billed as a standalone "da pagare" charge (NOT a lesson credit). Priced like a standard 60' lesson (`limits.lessonPrice60`) — no dedicated price setting.
- **License-aware**: when the Vehicles module is on, only students whose pursued license matches the lesson vehicle (category + transmission) can be pre-added/invited/accept (`vehicleServesLicense`).

## Data model (`prisma/schema.prisma`)
- **`AutoscuolaGroupLesson`** (container): `id, companyId, instructorId?, vehicleId?, startsAt, endsAt?, capacity (default 3), status("scheduled"|"cancelled"), priceAmount, notes, createdByUserId`. **Capacity is user-configurable (3 or 4) since 2026-06-12**: selectable at creation (web dialog + mobile sheet) and editable later (web manage dialog + mobile manage sheet, `updateGroupLesson({capacity})` — refused below the current enrolled count). The agenda bootstrap annotates every group-lesson row (incl. `gl-empty:` synthetic rows) with `groupLessonCapacity` so mobile seat dots show the real capacity.
- **`AutoscuolaGroupLessonInvite`** + **`AutoscuolaGroupLessonInviteResponse`** — mirror `AutoscuolaWaitlistOffer`/`...Response`. Status: invite `broadcasted`→`filled`/`cancelled`; response `accepted`/`declined`. `@@unique([inviteId, studentId])` blocks double-accept.
- **`AutoscuolaAppointment.groupLessonId`** (nullable FK, `onDelete: Cascade`): participants are real appointment rows with `type="group_lesson"` → they reuse agenda/overlap/payments/history for free.
- **`CompanyMember.groupLessonsOptIn Boolean @default(false)`**.
- Migration: `20260610120000_add_group_lessons` (additive, no backfill).

## Payment semantics
Participant appointments are written **directly** (NOT via `prepareAppointmentPaymentSnapshot`, which would consume a credit): `creditApplied=false, paymentRequired=true, paymentStatus="pending", manualPaymentStatus="unpaid", priceAmount=getGroupLessonPrice()`. `paymentStatus="pending"` is **inert to Stripe** (the auto-charge job only touches `pending_penalty`/`partial_paid`); the charge surfaces in the payments list (`paymentRequired=true`) and as a manual "da pagare" item. `getGroupLessonPrice` is in `lib/autoscuole/payments.ts`.

## Server actions (`lib/actions/autoscuole.actions.ts`)
- `createGroupLesson({startsAt, endsAt, vehicleId?, instructorId?, capacity?, studentIds?, notes?})` — OWNER+INSTRUCTOR (instructor auto-assigns self). Gated on `groupLessonsEnabled`. Validates instructor/vehicle, students (opted-in + license), overlap; transaction creates the container + one participant appointment per pre-added student.
- `addGroupLessonParticipant` / `removeGroupLessonParticipant` (cancels the seat + drops its pending charge).
- `cancelGroupLesson` — cancels the container, all participant appointments, and open invites.
- `getGroupLessonsForAgenda({from?, to?})` — returns lessons with `filledSeats`/`openSeats`/`participants`, incl. **empty** lessons (used by `GET /api/autoscuole/group-lessons` list). Empty-lesson agenda cards now come from the bootstrap `gl-empty:` synthesis, not this action.
- `getGroupLesson(id)` — single lesson detail (instructor/vehicle/seats/participants). Consumed by the mobile "Gestisci guida di gruppo" modal AND the student "Ritira iscrizione" sheet. Each participant now carries its **per-student note** (`participants[].notes`, the seat appointment's note) for the roster note editor; surfaced to staff and to the student's own seat (anonymous seats return `notes: null`). **Student-safe**: a non-staff caller (STUDENT) may only read a lesson they're enrolled in, and the other participants are returned **anonymous** (`appointmentId`/`studentId` masked to `""`, `studentName: null`) so seat counts work without leaking identities.
- `updateGroupLesson({groupLessonId, startsAt?, endsAt?, instructorId?, vehicleId?, notes?})` — edits time/instructor/vehicle and **cascades to all participant appointments**. ⚠️ The `notes` param (container note) is **deprecated from the UX** (2026-06-16): notes are now **per-student** (see below). No UI sets the container note anymore.
- `listEligibleGroupLessonInvitees(groupLessonId)` — opted-in, not-enrolled, license-compatible students (needs an existing lesson).
- `listOptedInGroupLessonStudents()` — opted-in students + license info **without** an existing lesson; used by the web **create** dialog to pre-add eligible students. OWNER/INSTRUCTOR/admin gated.
- `updateStudentGroupLessonOptIn({studentId, optIn})` — OWNER/admin OR any instructor.
- `getAutoscuolaAgendaBootstrapAction` now also returns `groupLessonsEnabled` (next to `vehiclesEnabled`, from `CompanyService.limits`) so the web agenda knows whether to show the "Nuovo → Guida di gruppo" item.
- Helpers: `validateGroupLessonStudents`, `findGroupLessonOverlap`, `GROUP_LESSON_ACTIVE_STATUSES`, `GROUP_LESSON_ENROLLED_STATUSES`.

## Per-student notes (2026-06-16)
The instructor takes a **separate note for each participant** (no more single container note). Each participant is an `AutoscuolaAppointment`, so the note simply lives on the **seat's own `notes` field** — no new model, no migration.
- **Seats start empty**: `createGroupLesson` and `addGroupLessonParticipant` set seat `notes: null` (they no longer copy the container note).
- **Editing**: reuse `updateAutoscuolaAppointmentDetails({ appointmentId, notes })` (the standard appointment-notes action — already authorises the owning instructor/owner, even on concluded lessons). Web: `GroupLessonManageDialog` adds a per-participant note editor (textarea). Mobile instructor: the roster `manage-group-lesson-participants` opens the shared `notesEditorStore` per student.
- **Student sees it**: the seat note flows through the existing `getAppointments`/`latest-note` paths, so it appears in the mobile **"Le mie note"** (`StudentMyNotesScreen`) with a dedicated **teal group-lesson card** (type `group_lesson`). No new endpoint.

## Past-lesson roster fix (`GROUP_LESSON_ENROLLED_STATUSES`, 2026-06-16, esteso 2026-07-15)
`GROUP_LESSON_ACTIVE_STATUSES` (`scheduled/confirmed/proposal/checked_in`) **excludes the past/terminal statuses**, so once a past group lesson's time passed its seats vanished from roster/count queries → managing a **past** group lesson showed 0 participants (and let you re-add already-enrolled students). Fix: new `GROUP_LESSON_ENROLLED_STATUSES` (= ACTIVE + `pending_review` + `completed` + `no_show`) applied to the **3 per-lesson roster/count queries**: `getGroupLesson`, `getGroupLessonsForAgenda`, `listEligibleGroupLessonInvitees` (enrolled set). The booking/eligibility paths (and all of `autoscuole-availability.actions.ts`, which only acts on still-`scheduled` future lessons) keep `ACTIVE` — these past statuses never occur there, so no behavioural change.

**`pending_review` aggiunto 2026-07-15 (secondo bug Robatto):** il cron `processAutoscuolaAutoPendingReview` (`autoscuole-reminders`, ogni minuto) porta le seat `scheduled/confirmed` con `endsAt ≤ now` a **`pending_review`** ("Da confermare") — il PRIMO stato in cui una guida passata atterra, PRIMA di `completed`/`no_show` (che richiedono una conferma manuale, assente se l'autoscuola non ha l'auto-checkin). Mancando da ENROLLED, ogni guida di gruppo passata-ma-non-revisionata mostrava **0/N nel roster** mentre l'agenda mostrava **N/N** (conta le righe grezze). Guide del 14/07 a Robatto: 3 seat `pending_review` = il "3/3", roster a 0.

## Presenze partecipanti (present/absent, 2026-07-15)
Ogni seat è un `AutoscuolaAppointment`, quindi la presenza vive sul suo `status`. Due azioni **dedicate** (NON `updateAutoscuolaAppointmentStatus`, che pretende un tipo-guida valido e innescherebbe credito/settlement, sbagliati per una seat flat "da pagare"):
- `setGroupLessonSeatOutcome({ appointmentId, outcome: "present"|"absent" })` — present = `completed` se la guida è finita, altrimenti `checked_in`; absent = `no_show`. **Nessun limite temporale**: correggibile presente↔assente **anche a guida passata** (unica guardia: seat non `cancelled` + istruttore solo sulle proprie guide). **Non tocca la fatturazione** (il posto resta "da pagare" comunque — nessuna penale no-show come sulle guide singole): è puro registro presenze.
- `markGroupLessonAllPresent({ groupLessonId })` — segna tutte le seat non annullate come presenti in un colpo.
- `getGroupLesson` espone `participants[].attendance` (`"present"|"absent"|"pending"`, helper `groupLessonAttendance`). `"pending"` = upcoming o past-but-unreviewed (`pending_review`).
- **API mobile**: `POST /api/autoscuole/group-lessons/[id]/attendance` — body `{ all: true }` (tutti presenti) o `{ appointmentId, outcome }` (singola).
- **UI** (tint-aware: teal auto / arancio moto, "Assente" grigio neutro): web `GroupLessonManageDialog` (pill sotto ogni allievo + "Segna tutti presenti" nell'header); mobile `manage-group-lesson-participants` (stessa pill + scorciatoia, spinner→refetch, no optimistic). Roster ordinato `createdAt asc` (`getGroupLesson`/`getGroupLessonsForAgenda`) così non si rimescola al cambio esito; il web patcha lo stato locale (niente reload/flash), il mobile congela l'ordine mostrato (`keepOrder`).

## Invite actions (`lib/actions/autoscuole-availability.actions.ts`)
- `broadcastGroupLessonInvite({companyId, groupLessonId, expiresAt})` — recipients = opted-in + license-compatible + available + no conflict + not enrolled; push `data.kind:"group_lesson_invite"` (also email/whatsapp per `slotFillChannels`).
- `inviteToGroupLesson({groupLessonId, expiresInHours?})` — public wrapper (OWNER+INSTRUCTOR); expiry = min(now+hours, lesson start), default 24h.
- `respondGroupLessonInvite({inviteId, studentId, response})` — **multi-seat accept** in a transaction guarded by `SELECT … FOR UPDATE` on the lesson row + the `[inviteId, studentId]` unique constraint → no overbooking. Creates the participant appointment (da pagare); marks the invite `filled` when the last seat is taken.
- `getGroupLessonInvites({studentId, limit?, countOnly?})` — **lesson-first discovery** for the mobile "Guide di gruppo" student list (NOT invite-row-first, fix 2026-06-15 after the Robatto bug where only 0-occupied lessons were visible). **Limit cap 100, era 20 (2026-07-06, secondo bug Robatto)**: con 3-4 guide/giorno il cap a 20 nascondeva tutte le lezioni oltre ~2 settimane ("vedo le guide di gruppo solo fino al 23/07"); la lista mobile ora chiede 100. **`countOnly`** (badge home allievo): stessa lista con `inviteId: null`, salta le scritture `ensureDiscoverableGroupLessonInvite` — il badge usa solo `.length`. Iterates ALL scheduled, future, non-full group lessons of the company and applies the same eligibility as the broadcaster (opted-in + license-compatible + no time conflict + not already enrolled), then for each surfaced lesson **lazily ensures a discovery invite handle** via `ensureDiscoverableGroupLessonInvite` (no push/email/whatsapp side-effects) so the existing `respondGroupLessonInvite(inviteId, …)` accept flow keeps working unchanged. Rationale: the invite row is only the push **nudge** (single live invite per lesson, `expiresAt = min(start, now+24h)`); discovery must NOT be gated on it, otherwise a partially-filled lesson (or one whose 24h nudge lapsed, or one that never got an invite) silently drops out of the student list despite open seats. **Declines stick**: lessons the student has a `declined` response on (checked across ALL of the lesson's invites, since invite rows churn) are excluded — so "Non mi interessa" hides the lesson permanently and the home count drops (fix 2026-06-15). Accepting excludes the lesson via enrollment instead. **Returns `kind`** (`"standard"|"moto"`, added 2026-06-26) so the mobile invite card can say "ti verrà assegnata una moto" — the specific moto is auto-assigned only at acceptance, so a moto invite has no concrete `vehicleName` to show.
- `ensureDiscoverableGroupLessonInvite(companyId, groupLessonId, studentId, startsAt)` — internal, no-notification helper: reuses the latest live `broadcasted` invite (refreshing `expiresAt`→lesson start if it had expired) when this student hasn't answered it; otherwise supersedes it and issues a fresh handle. Guarantees the returned inviteId is `broadcasted`, unexpired, and free of a prior response from this student (so "Iscrivimi" never hits the `[inviteId, studentId]` "Hai già risposto" guard).
- `cancelGroupLessonParticipantAppointment({companyId, appointmentId, actorUserId})` — **shared cancellation helper** (single source of truth) used by instructor removal AND student withdrawal AND the defensive `cancelAutoscuolaAppointment` delegation. Payment behaviour mirrors a normal lesson (decision **A**, 2026-06-10): **before `penaltyCutoffAt`** → frees the seat and zeroes the charge (`paymentRequired:false`, `not_required`, `manualPaymentStatus:null`); **after the cutoff or after the lesson happened** → keeps it "da pagare" and lets the existing **late-cancellations inbox** (`getLateCancellations`/`resolveLateCancellation`) pick it up via `cancellationKind:'manual_cancel'` + `penaltyCutoffAt` (no type filter there). Then, if the lesson is still scheduled/future with open seats, **re-broadcasts** an invite (`broadcastGroupLessonInvite`). If the actor is the student themselves, **pushes the instructor** a `appointment_cancelled` notification.
- `withdrawFromGroupLesson({groupLessonId})` — STUDENT withdraws their own seat (resolves their own active appointment, guards "lesson not started", delegates to the shared helper).

> **Prerequisite (penalty fields at creation):** all 3 seat-creation sites (`createGroupLesson`, `addGroupLessonParticipant`, `respondGroupLessonInvite`) now set `penaltyCutoffAt` (= `startsAt − penaltyCutoffHours`) and `penaltyAmount` via `getGroupLessonPenaltySnapshot` (`lib/autoscuole/payments.ts`), so the early/late distinction and the late-cancellations inbox work. **No migration** — these columns already exist on `AutoscuolaAppointment`.

## Behaviour notes
- Capacity is fixed at 3 in the mobile create flow (the field exists on the model for flexibility).
- No `trigger:deploy` is required unless an invite-expiry cron is added (v1 = passive expiry checked at respond/list time via `expiresAt`). The empty-slot notification cron is unrelated.

## Participation behaviour (decided 2026-06-10, implemented)
- **Student withdrawal**: YES. The student frees their seat → it's **re-broadcast** to eligible students. Instructor gets a push.
- **Check-in**: none. The instructor may **remove** a participant any time — even close to the lesson and **even after** it happened — and the removal flows into late cancellations like a normal lesson.
- **Payment (decision A)**: reuse the **same normal-lesson cancellation + late-cancellation logic** — early cancel = seat freed, no charge; late cancel / removal = stays "da pagare" and surfaces in the *cancellazioni tardive* inbox (owner charges/dismisses). The old "always zero the charge" behaviour of `removeGroupLessonParticipant` is gone.

## Moto group lessons (`kind="moto"`, 2026-06-25)
A **second flavour** of group lesson for motorcycles: instead of one shared vehicle, the instructor reserves a **fleet of motos** (chosen by hand) plus **one shared follow car** ("auto al seguito"), and **each participant rides their own moto** — categories may be **mixed** (A1/A2/A together). When a student enrols the system **auto-assigns** a still-free fleet moto matching their license. Eligibility is therefore **dynamic** (a compatible moto must remain). Standard groups are unchanged (`kind="standard"`).

- **Schema** (`20260625130000_group_moto_lessons`, additive): `AutoscuolaGroupLesson.kind ("standard"|"moto")`, `AutoscuolaGroupLesson.followVehicleId?` (shared follow car, category B), and **`AutoscuolaGroupLessonVehicle (groupLessonId, vehicleId)`** = the moto fleet. For `kind="moto"` the container `vehicleId` is null; each participant appointment carries its **own** moto in `vehicleId` + an `AutoscuolaAppointmentVehicle role="primary"` row. The follow car is reserved **only at the container level** — never on participant join rows (else the conflict check would block the 2nd/3rd participant).
- **Decision logic** (`lib/autoscuole/group-moto.ts`, pure + unit-tested): `assignMotoForStudent` / `assignMotosToStudents` (pick a free compatible fleet moto), `eligibleForMotoGroup` (dynamic eligibility), `groupMotoFollowCarRequired` + `validateMotoGroupSetup` (fleet/follow-car validation, capacity ≤ fleet). Reuses `vehicleServesLicense`, `isMotoLicenseCategory`, `requiresFollowCar`, `FOLLOW_CAR_CATEGORY`.
- **Capacity** = the chosen value, **capped at the fleet size** (the real seat count is "how many motos").
- **Follow car — SEMPRE facoltativa alla creazione (2026-07-06)**: si può scegliere a mano, ma se manca NON blocca più (rimosso `follow_car_required_missing`). Quando `followCarRules` la richiede per una categoria della flotta, viene **auto-assegnata SOLO al PRIMO iscritto** di una guida senza auto (pre-add in creazione, add staff, accept invito con 0 riders) da `findFreeGroupFollowCar` (`lib/autoscuole/group-follow-assign.ts`): prima auto B attiva accessibile all'istruttore e libera nell'intera finestra (no appuntamenti diretti/join, no altri container). Nessuna libera → l'iscrizione fallisce con messaggio chiaro. **La scelta esplicita dello staff vince sempre**: togliere l'auto dal dialog/sheet di gestione ("Nessuna") persiste — né l'update né le iscrizioni successive la riassegnano (una guida con riders e senza auto = decisione dello staff).
- **Reservations / conflicts**: `lib/autoscuole/group-lesson-busy.ts` now reserves the **follow car + every fleet moto** for the whole window (even before seats fill). `findGroupLessonOverlap` takes a `vehicleIds[]` set and also checks other **group-lesson containers** (their `vehicleId`/`followVehicleId`/fleet), closing the gap where empty containers were invisible.
- **Server actions**: `createGroupLesson`/`updateGroupLesson` accept `kind`, `vehicleIds` (fleet), `followVehicleId`. Enrolment (`addGroupLessonParticipant`, `respondGroupLessonInvite`) **auto-assigns** a moto in a `FOR UPDATE`-serialized transaction. `updateGroupLesson` for moto does **not** cascade a vehicle onto participants (each keeps its moto) and refuses dropping a moto still assigned to a participant. `listEligibleGroupLessonInvitees` filters by dynamic moto eligibility. Read shapes (`getGroupLesson`, `getGroupLessonsForAgenda`) expose `kind`, `followVehicle*`, `fleet`, and each participant's assigned moto.
- **Web UI**: `GroupLessonCreateDialog` has a Standard/Moto toggle, a moto-fleet multi-select and a follow-car picker; `GroupLessonManageDialog` shows the per-participant moto + a read-only fleet/follow-car summary (no vehicle cascade). Mobile: `CreateGroupLessonScreen` mirrors it (segmented toggle + multi-picker fleet + follow car).

- **Solo auto per i gruppi standard (2026-07-07)**: i picker Veicolo dei gruppi standard (create + manage, web e mobile) escludono le moto, e il BE rifiuta una moto come veicolo condiviso di un gruppo standard ("usa la modalità Moto") sia in creazione sia in modifica.
